# FitVibe Glossary

This glossary defines terminology, acronyms, and domain-specific terms used throughout the FitVibe documentation and codebase.

## A

- **2FA**: Two-Factor Authentication - a second verification step (e.g., TOTP, security key) in addition to password.
- **AC**: Acceptance Criteria - testable conditions that must be met for a requirement to be considered complete.
- **ACL**: Access Control List - a list of permissions attached to a resource that specifies which users or system processes are granted access.
- **ACME**: Automated Certificate Management Environment - protocol used by Let's Encrypt for automated certificate issuance and renewal.
- **Adapter**: A design pattern that allows incompatible interfaces to work together; external services (e.g., email) are adapterized to be mockable in tests.
- **ADR**: Architecture Decision Record - structured documentation of key technical decisions.
- **APM**: Application Performance Monitoring - tools and processes for monitoring application performance in production.
- **AV Scan**: Antivirus scanning - automated security scanning of uploaded files to detect malware before storage.

## B

- **BE**: Backend - server-side application layer (see also FE).

## C

- **CI/CD**: Continuous Integration / Continuous Deployment - automated pipelines for building, testing, and deploying code.
- **CLS**: Cumulative Layout Shift - layout stability measure for web performance.
- **CORS**: Cross-Origin Resource Sharing - HTTP mechanism that allows web pages to access resources from a different domain.
- **cosign**: A tool for signing and verifying container images and other artifacts.
- **CSP**: Content Security Policy - HTTP header that restricts allowed sources of content to mitigate XSS and data exfiltration.
- **CRUD**: Create, Read, Update, Delete - the four basic operations for persistent storage.
- **CSRF**: Cross-Site Request Forgery - an attack that forces authenticated users to submit requests to a web application.

## D

- **DB**: Database - persistent data store (PostgreSQL in FitVibe).
- **DPIA**: Data Protection Impact Assessment - a process to identify and minimize data protection risks.
- **DR**: Disaster Recovery - processes and capabilities to restore service after major incidents.
- **DSR**: Data Subject Rights - GDPR rights (access, rectification, deletion, portability).
- **DTO**: Data Transfer Object - an object that carries data between processes or layers, typically used in API contracts.

## E

- **EICAR**: European Institute for Computer Antivirus Research - standard test file used to verify antivirus software is working correctly.
- **ERD**: Entity-Relationship Diagram - a visual representation of database structure showing entities and their relationships.
- **Error Budget**: The amount of error rate that a service can consume over a period of time before violating SLOs.
- **Ephemeral**: Temporary or short-lived, often referring to test databases or containers that are created and destroyed for each test run.

## F

- **FakeClock**: A test utility that allows control over time in tests, enabling deterministic testing of time-dependent code.
- **FE**: Frontend - client-side application layer (see also BE).
- **FID**: First Input Delay - interaction responsiveness metric for web performance.
- **Flaky Test**: A test that produces inconsistent results (pass/fail) on the same code commit without deterministic cause.
- **FR**: Functional Requirement - specifies what the system shall do.

## G

- **GDPR**: General Data Protection Regulation - EU regulation governing data protection and privacy.
- **GHCR**: GitHub Container Registry - container image registry provided by GitHub.
- **Golden Response**: A reference response stored in tests to compare against actual responses, ensuring consistency.

## H

- **HSTS**: HTTP Strict Transport Security - forces browsers to use HTTPS for a domain.

## I

- **i18n**: Internationalization - the process of designing software to support multiple languages and locales.
- **Idempotency**: Property of an operation that can be applied multiple times without changing the result beyond the initial application.
- **Idempotency Key**: A unique identifier sent with state-changing requests to prevent duplicate processing.

## J

- **Jest**: JavaScript testing framework commonly used for unit and integration tests.
- **JWT**: JSON Web Token - token format used for authentication.

## K

- **k6**: Load testing tool for testing API performance and scalability.
- **KPI**: Key Performance Indicator - measurable target used to assess performance goals.

## L

- **LCP**: Largest Contentful Paint - metric for perceived load speed.
- **Let's Encrypt**: Free, automated certificate authority providing TLS certificates.
- **Lighthouse**: Google's automated tool for auditing web page quality, performance, and accessibility.
- **Locale**: A set of parameters that defines the user's language, region, and cultural preferences (e.g., `en-US`, `de-DE`).

## M

- **MailHog**: Email testing tool that captures emails sent during development.
- **Materialized View**: A database object that contains the results of a query, stored for performance optimization.
- **MVP**: Minimum Viable Product - smallest feature set that delivers value to users.

## N

- **NGINX**: Web server and reverse proxy used for TLS termination, rate limiting, and security headers.
- **NFR**: Non-Functional Requirement - specifies system qualities and constraints.

## O

- **OCSP**: Online Certificate Status Protocol - mechanism to check TLS certificate revocation (often via stapling).
- **OpenAPI**: Specification for describing RESTful APIs, formerly known as Swagger.
- **OpenTelemetry (OTEL)**: Open standard for observability, providing APIs and SDKs for collecting telemetry data.
- **OTLP**: OpenTelemetry Protocol - the protocol used by OpenTelemetry to export telemetry data.
- **OWASP**: Open Worldwide Application Security Project - community and standards body behind the "OWASP Top 10".

## P

- **p95**: 95th percentile measurement for latency or duration metrics.
- **Partition**: Database partitioning - splitting large tables into smaller, more manageable pieces for performance and maintenance.
- **PII**: Personally Identifiable Information - data that can identify a person (e.g., name, email, IP, device ID).
- **Playwright**: End-to-end testing framework for web applications.
- **PRD**: Product Requirements Document - specifies which features the system entails.
- **PRNG**: Pseudo-Random Number Generator - deterministic algorithm that generates sequences of numbers that approximate random sequences, used in tests for reproducibility.

## Q

- **QA**: Quality Assurance - the process of ensuring software quality through testing and validation.

## R

- **Rate Limiting**: Controlling the rate of requests from clients to prevent abuse and ensure fair resource usage.
- **RBAC**: Role-Based Access Control - access control method where permissions are assigned to roles, and users are assigned roles.
- **REST**: Representational State Transfer - architectural style for designing web services.
- **RPE**: Rate of Perceived Exertion - subjective measure of exercise intensity (typically 1-10 scale).
- **RPO**: Recovery Point Objective - maximum acceptable data loss window in disaster recovery.
- **RS256**: RSA signature algorithm with SHA-256 - asymmetric algorithm used for JWT signing in FitVibe.
- **RTM**: Requirements Traceability Matrix - a document that maps requirements to tests and evidence.
- **RTO**: Recovery Time Objective - maximum acceptable downtime for recovery.
- **RUM**: Real User Monitoring - frontend performance telemetry collected from real users.

## S

- **SBOM**: Software Bill of Materials - machine-readable inventory of components in a software artifact.
- **SCA**: Software Composition Analysis - dependency scanning to detect vulnerabilities and license issues.
- **Seeded**: Pre-populated with test data, often referring to databases or test fixtures.
- **SLI**: Service Level Indicator - a quantitative measure of service quality (e.g., availability, latency).
- **SLO**: Service Level Objective - agreed target level for service availability or performance.
- **SMTP**: Simple Mail Transfer Protocol - standard protocol for email transmission.
- **SSOT**: Single Source of Truth - principle that data should be stored in one authoritative location.
- **Supertest**: HTTP assertion library for testing Node.js HTTP servers.
- **Swagger**: See OpenAPI.

## T

- **TDD**: Technical Design Document - specifies how the system will implement features and constraints.
- **Test Double**: Generic term for any object used in place of a real object during testing (includes mocks, stubs, spies, fakes).
- **TLS**: Transport Layer Security - protocol for encrypted communications on the network (v1.3 referenced).
- **TOTP**: Time-based One-Time Password - algorithm for two-factor authentication.
- **Transactional**: Database operations that are atomic - either all succeed or all fail (ACID properties).
- **TTI**: Time-to-Interactive - time until the page becomes reliably interactive for user input.

## U

- **ULID**: Universally Unique Lexicographically Sortable Identifier - alternative to UUID that is time-ordered.
- **UUIDv7**: UUID version 7 - time-ordered UUID format that provides better database indexing performance than UUIDv4.

## V

- **Vitest**: Fast unit test framework for Vite-based projects.

## W

- **Waiver**: Formal exception to a quality gate or requirement, requiring documented rationale and approval.
- **WCAG**: Web Content Accessibility Guidelines - international standard for digital accessibility (v2.1 AA referenced).

## X

- **XSS**: Cross-Site Scripting - security vulnerability that allows attackers to inject malicious scripts into web pages.

## Z

- **ZAP**: OWASP ZAP (Zed Attack Proxy) - security testing tool for finding vulnerabilities in web applications.
- **Zod**: TypeScript-first schema validation library used for runtime type checking and validation.
